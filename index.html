Step 2 — Gameplay Events
Stage 2

Gameplay Events

The game engine emits one event per task attempt. These raw events are the only data the ML system ever sees — no video, no audio, no camera.

Live Event Stream
Each event fires when a child completes one task — a match, a sort, a trace. The system accumulates a rolling window of the last 20 events for feature extraction.

Step 4 — ML Model 1
Stage 4 — Machine Learning

Model 1: Skill Mastery Classifier

11 independent XGBoost classifiers — one per skill. Each answers the same question for its skill: "What is the probability this child has mastered this skill right now?"

🧠

Architecture

Why 11 separate models instead of one?

Independence of Evidence
A child can be great at Color Sorting (visual) but terrible at Phonics (auditory). One combined model would let counting performance corrupt phonics predictions. Separate models = clean estimates.
Cold Start Per Skill
Each skill has its own data sparsity. A child may have 50 counting events but 3 phonics events. Each model handles its own cold start with age-band priors independently.
Calibration Per Skill
Isotonic calibration is fitted separately per skill. A threshold of 0.70 means "70% probability mastered" — and that is actually true because calibration enforces it.
⚙️

XGBoost Configuration

Step 5 — ML Model 2
Stage 5 — Machine Learning

Model 2: Engagement & Fatigue Classifier

One XGBoost classifier with 3 output classes: Engaged, Neutral, and Fatigued. It reads the current session temperature — completely separate from skill ability.

🔥

Why a Separate Model?

Ability ≠ State. A child with high counting mastery (Model 1) can still be exhausted right now (Model 2). Mixing them would corrupt both. Model 1 measures what the child can do. Model 2 measures how the child is doing right now.

Step 6 — Formula Engine
Stage 6 — Deterministic

Formula Engine

The ML models never make product decisions directly. The Formula Engine reads their outputs and applies transparent, auditable rules to choose the next action. Zero ML here.

🔑
The Core Principle
ML tells you the state of the world. Business logic decides what to do about it. Never let a model make a product decision directly — that's how you get unsafe, unexplainable outcomes in a child-facing product.
